Backend project for **Lyringo**.

- Product name: `Lyringo`

## Architecture

This project uses a **modular monolith** with **Clean Architecture / Hexagonal Architecture**.

Current modules:

```text
com.lyringo.auth
com.lyringo.users
com.lyringo.music
com.lyringo.shared
```

Each business module should be organized as:

```text
module/
  domain/            # Entities, value objects, domain rules. No Spring/JPA.
  application/       # Use cases, commands, DTOs, ports. No Spring/JPA/HTTP.
  infrastructure/    # JPA, Redis, S3, JWT, BCrypt, external services.
  interfaceadapter/  # REST controllers, GraphQL resolvers, request/response mapping.
```

Dependency rule:

```text
interfaceadapter -> application -> domain
infrastructure -> application/domain
```

The `domain` and `application` layers must not depend on Spring, JPA, Servlet APIs, or transport concerns.
This rule is enforced by ArchUnit tests in `src/test/java/com/lyringo/architecture`.

## Requirements

- Java 21+
- Maven 3.9+
- Docker, for local PostgreSQL and Redis

## Run local infrastructure

```bash
docker compose up -d postgres redis
```

## Database migrations

Database schema changes are managed by Flyway, not Hibernate auto DDL.

- Keep `spring.jpa.hibernate.ddl-auto=validate` so Hibernate only checks the schema.
- Put SQL migrations in `src/main/resources/db/migration`.
- Name new migrations like `V2__add_user_profile_fields.sql`.
- Do not edit an old migration after it has run in a shared database; add a new versioned migration instead.

## SonarQube Cloud

The repository includes `.github/workflows/sonar.yml`.

Before enabling it, create the GitHub repository secret:

```text
SONAR_TOKEN
```

Then update these values in `.github/workflows/sonar.yml`:

```text
-Dsonar.projectKey=lyringo_lyringo-backend
-Dsonar.organization=lyringo
```

Use the actual project key and organization generated by SonarQube Cloud.
